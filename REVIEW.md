## Загальний опис

**nintendoflow** — Telegram-бот, що автоматично збирає Nintendo-новини з RSS, оцінює релевантність за ключовими словами, переписує статті через AI та публікує в канал. Запускається через GitHub Actions (6 разів/день), вебхук-коллбеки модерації обробляє Vercel Go function.

## 🔴 КРИТИЧНІ ПОМИЛКИ

### 1. `cron.yml` — продакшн працює в режимі модерації

**Файл:** `.github/workflows/cron.yml`, рядки 33–38

```yaml
TEST_TELEGRAM_TOKEN: ${{ secrets.TELEGRAM_BOT_TOKEN }}   # той самий токен!
TEST_CHANNEL_ID: ...
TEST_ADMIN_CHAT_ID: ...
TEST_MODERATION_MODE: "true"                              # ВМИКАЄ МОДЕРАЦІЮ У ПРОД
```

**Наслідок:** Бот не публікує пости напряму — він надсилає їх адміну на підтвердження замість публікації в канал. Якщо адмін не натискає кнопку, контент не виходить взагалі.

**Виправлення:**
- Створити окремий секрет `TEST_TELEGRAM_TOKEN` лише для тестового workflow.
- Прибрати `TEST_MODERATION_MODE: "true"` з `cron.yml` (або встановити `"false"`).
- `cron-test.yml` — залишити як є, він коректний.

### 2. Вебхук падає, якщо `TELEGRAM_WEBHOOK_SECRET` не задано

**Файл:** `api/webhook.go`, ~рядок 61

```go
if secret == "" {
    http.Error(w, "forbidden", http.StatusForbidden)
    return
}
```

Якщо на Vercel забули виставити env-змінну — всі вебхуки повертають 403. Немає ні попередження в логах, ні fallback.

**Виправлення:** Під час старту Vercel function (або в init) логувати `slog.Error("TELEGRAM_WEBHOOK_SECRET is not set")` та негайно повертати помилку з розгорнутим повідомленням у логи, щоб проблема була очевидна.

## 🟠 СЕРЙОЗНІ ПРОБЛЕМИ

### 3. Крихкий парсинг затримки rate-limit у AI manager

**Файл:** `pkg/ai/manager.go`, рядки 174–191

Три різних regex для вилучення затримки з повідомлення помилки:
```
"retry after (\d+)"
"wait (\d+) second"
"in (\d+\.?\d*) seconds"
```

Якщо провайдер змінить формат відповіді (наприклад Groq чи OpenRouter) — код мовчки падає на дефолтну затримку 20с, і ви не дізнаєтесь про це. Регулярки чутливі до регістру.

**Виправлення:** Додати логування `slog.Warn("could not parse retry delay, using default")` + зробити regex case-insensitive (`(?i)`).

### 4. Race condition при оновленні OAuth-токена Reddit

**Файл:** `pkg/fetcher/reddit.go`, рядки 186–199

Глобальний стан `redditToken` / `redditTokenExpires` без mutex. Якщо два goroutine одночасно перевіряють термін дії та обидва вирішують оновити — виконуються два HTTP-запити, один перезаписує інший.

**Виправлення:**
```go
var redditMu sync.Mutex
// оборнути блок оновлення в redditMu.Lock() / Unlock()
```

### 5. Непов'язаний ліміт з'єднань у Vercel webhook

**Файл:** `api/webhook.go`, ~рядок 37

```go
db.SetMaxOpenConns(3)
```

Vercel запускає кожен запит в окремому інстансі (cold start). `SetMaxOpenConns(3)` на рівні пакету нічого не обмежує між запитами — з'єднання можуть накопичуватись у Neon під навантаженням.

**Виправлення:** Додати `db.SetConnMaxLifetime(5 * time.Minute)` та `db.SetMaxIdleConns(1)`.

---

### 6. Відсутній таймаут на скачування YouTube-відео

**Файл:** `pkg/telegram/youtube.go` + `cmd/bot/main.go`, ~рядок 283

Таймаут в 10с задано на HTTP-запит метаданих, але стрімінг бінарного відео (до 49 МБ) не має окремого обмеження по часу і розміру тіла.

**Виправлення:**
```go
resp.Body = http.MaxBytesReader(nil, resp.Body, 50<<20) // 50 MB hard limit
```
Та окремий `context.WithTimeout` на download.

## 🟡 СЕРЕДНІ ПРОБЛЕМИ

### 7. Невідповідність порогу дедуплікації

**Файл:** `pkg/dedup/dedup.go`, рядок 90 vs `cmd/bot/main.go`, рядок 188

`IsDuplicate()` всередині хардкодить поріг 0.65, але в `main.go` викликається `ThresholdForSourceType()` яке повертає різні значення (0.55 для агрегаторів, 0.78 для офіційних). Одна з гілок дедупу ігнорує цей поріг.

**Виправлення:** Передавати поріг параметром у `IsDuplicate(threshold float64)` і прибрати хардкод.

---

### 8. Неповна санітизація prompt injection

**Файл:** `pkg/ai/prompt.go`, рядки 17–26

Фільтруються лише 8 фраз (`"Ignore previous"`, `"[INST]"` і т.д.). Відсутні:
- `"SYSTEM:"`, `"DEVELOPER:"`, `"RULES:"`
- Markdown-блоки ` ```system ``` `
- Unicode lookalikes (кирилиця замість латиниці у командах)

**Виправлення:** Розширити список + додати нормалізацію unicode перед перевіркою.

---

### 9. Відсутнє версіонування міграцій

**Файл:** `migrations/001–006_*.sql`

Міграції застосовуються по черзі без таблиці версій (`schema_migrations`). Якщо бот і export-job запустяться одночасно, або міграція зафейлить на середині — немає механізму перевірки, що саме вже застосовано.

**Виправлення:** Додати `migrations` таблицю або використати `golang-migrate`.

### 11. Cleanup edit-сесій не гарантований

**Файл:** `api/webhook.go`, рядки 187–188, 203–204

Сесія видаляється після дії (Publish/Reject/Edit), але:
- Якщо вебхук крашнеться між оновленням статусу і видаленням сесії — сесія зависне.
- Немає фонового job для прибирання прострочених сесій.

**Виправлення:** Додати `DELETE FROM moderation_edit_sessions WHERE created_at < NOW() - INTERVAL '30 minutes'` на початку кожного вебхук-запиту.
